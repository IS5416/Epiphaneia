## 1. 概述

### 1.1 一句话定位

Epiphaneia 是面向后端开发者的 AI Agent 智能诊断工作台：用自然语言问"为什么 X 慢了/挂了？"，系统自动查询 Prometheus + 日志系统，由 LLM 完成根因推断，输出结构化诊断报告。**只读不写。**

### 1.2 问题陈述

开发者不缺监控工具，缺的是智能分析层。现状：告警只说"延迟高了"，定位根因靠人翻日志半小时；排查经验存在个人脑中，新人不会、老人重复劳动；商业 AIOps（PagerDuty/Datadog）价格将中小团队挡在门外，开源侧 Java 生态几乎空白。

### 1.3 核心公式

```
Epiphaneia ≠ 另一个监控工具
Epiphaneia = 监控工具的 AI 大脑层
           = 数据(已有监控) + 推理(LLM) + 经验(知识库)
```

### 1.4 产品边界总纲（继承 IdeaValidation 结论，不重议）

- **不采集、不存储监控数据**——站在 Prometheus/ELK 之上，零侵入
- **不做可视化 Dashboard**——不与 Grafana 竞争
- **不执行修复操作**——MVP 强制安全边界，架构层拦截，不依赖 prompt 约束
- **开源自部署**——B2D，docker-compose 一键启动是核心卖点

## 3. 用户故事

### 画像 A：后端开发者（主目标，P0 全覆盖）

- **US-A1**：作为后端开发者，我想用自然语言问"为什么 user-service 的 p99 从 200ms 涨到 2s"，让系统替我查指标和日志，这样我不用在 Grafana 和 Kibana 之间来回切换半小时。
- **US-A2**：作为后端开发者，我想得到一份含根因假设、证据链、修复建议、风险评估的报告，这样我可以直接贴进事故复盘文档。
- **US-A3**：作为后端开发者，我想让系统自动发现我的 Spring Boot Actuator 端点，这样我不用写配置文件。
- **US-A4**：作为后端开发者，我管着多个微服务（user-service / order-service / gateway），我想要在诊断工作台切换不同应用、各自有独立的诊断会话历史和 LLM 上下文，这样 A 应用的问题排查不会混入 B 应用的上下文。

### 画像 B：SRE/DevOps（P0 覆盖核心，v0.95 补全工作流）

- **US-B1**：作为 SRE，我想让 AlertManager 告警自动触发诊断并推送到飞书群，这样我被叫醒时看到的不是"延迟高了"而是"根因是 Redis 连接池耗尽，建议扩容 10→30"。（v0.95）
- **US-B2**：作为 SRE，我想让所有诊断能力通过 REST API 暴露，这样我可以接进现有工具链。
- **US-B3**：作为 SRE，我想让诊断报告带完整时间线和证据链，这样结论可追溯、可审计。

### 画像 C：Tech Lead（远期，本版本不承诺）

- **US-C1**：作为 Tech Lead，我想看过去 30 天服务健康度排行。（v1.0 知识库之后评估）

## 4. 功能需求

> 优先级体系沿用 `mvpScope.md`，不重新排序。P0 = v0.9 首发；P1 = v0.95/v1.0；P2 = v1.5+ 仅预留接口。
> 每项验收标准按 正常 / 异常 / 边界 三分——API 粒度的完整验收标准在 `apiDesign.md`。

### 4.1 P0 — v0.9 首发（缺一不发布）

#### FR-1 自然语言诊断问答

用户输入自然语言问题，Agent 自动完成：问题拆解 → 数据源查询规划 → 执行查询 → 综合分析 → 回答。多步推理过程对用户可见（见 FR-11）。

- 正常：输入"为什么 order-service 最近一小时报错变多？"，返回含根因假设的完整回答
- 异常：问题与运维无关（如"写首诗"），礼貌拒绝并说明能力范围；LLM 后端不可用时明确报错，不假装分析
- 边界：问题指向不存在的服务名，回答中明确"未找到该服务的指标/日志"并列出可用服务名候选

#### FR-2 Prometheus Connector

对接 Prometheus HTTP API：Agent 根据诊断需要自动生成 PromQL 查询，获取指标时序数据，识别异常区间。

- 正常：能查询 QPS、延迟分位、错误率、JVM 指标等常用指标并正确解析
- 异常：Prometheus 不可达/认证失败，诊断降级为"仅基于日志"并在报告中声明数据缺口
- 边界：查询区间无数据点、指标基数过大时截断并声明

#### FR-3 日志 Connector（Elasticsearch 首发）

基于时间窗口 + 关键词/服务名检索日志，提取错误模式（异常堆栈聚类、高频错误消息）。

- 正常：给定服务名和时间窗，返回错误日志样本与频次统计
- 异常：ES 不可达时降级为"仅基于指标"并声明
- 边界：命中日志量超限（如 >10,000 条）时采样并声明采样率
- 备注：Loki 支持走 Connector SPI，v1.0 或社区实现（见开放问题 OQ-1）

#### FR-4 LLM 根因推断

综合指标 + 日志 + 应用元信息（FR-6），输出结构化结论：根因假设（按置信度排序，最多 3 个）、证据链（每条假设引用具体指标/日志证据）、修复建议、风险评估。

- 正常：金标准故障集达标（G2）
- 异常：证据不足以支撑结论时，明确输出"证据不足"及需要补充的数据，禁止编造根因
- 边界：多故障叠加场景，允许输出多个并列假设而非强行归一

#### FR-5 诊断报告（会话级实时合成 + 导出即下载）

报告为**会话级实时合成**，不持久存储为独立实体。每轮问题 agent 回复含内联根因与证据摘要（mini 摘要），追问就在同一会话上下文中继续。点击"查看完整报告"时，LLM 读取完整会话上下文当场合成结构化 Markdown 报告（时间线、全部证据链、根因假设、修复建议、风险评估）。"导出"即浏览器下载 Markdown 文件，服务器不留存导出文件。报告语言跟随提问语言。

- 正常：会话级报告完整渲染，含全部 Q&A 证据链；导出文件格式正确
- 异常：会话为空时"查看报告"给出提示（"尚无诊断内容"）
- 边界：会话轮数增多导致 LLM 上下文窗口紧张时，截断策略：保留最后一轮完整 + 之前轮摘要（NFR-1 超时上限不变）。用户主动清空会话后，报告不可恢复——需保留的报告应在清空前导出。

> 设计决策：无独立 Report 表。Conversation 是聚合根，Message 是证据实体，报告是会话的只读视图。理由与备选方案评估见 `architectureDesignHandoff.md`。

#### FR-6 多应用管理 + Spring Boot 零配置接入

P6 设置页管理目标应用列表（应用名 + Actuator 地址 + 可选标签如 "prod" "staging"）。添加后自动探测 Actuator 端点，读取 health/metrics/env/info，作为该应用的诊断上下文。P3 诊断工作台顶部含应用切换器，切换应用 → 对应会话历史、LLM 上下文全部独立。**共用同一套 Prometheus + Elasticsearch 数据源**，数据隔离通过每个应用的 service label 自然达成。单实例单用户（多应用 ≠ 多租户），不涉及团队/权限/RBAC。

- 正常：标准 Spring Boot 3.x + Actuator 应用自动识别；多个应用间会话互不污染
- 异常：Actuator 未开启或端点受限时降级运行，提示用户可开启哪些端点获得更好诊断
- 边界：非 Spring Boot 应用仍可配置为"通用应用"（手动指定 Prometheus 查询 label），仅失去应用级上下文；应用数量上限暂不做硬限制（P50 诊断时长的隐式约束即可）

#### FR-7 Web UI

对话式诊断界面（类 ChatGPT 交互 + 顶部应用切换器）+ 历史会话列表 + 报告查看页 + 设置页（LLM / 数据源 / 多应用列表 / 安全）。UI 文案 English-first，预留 i18n。详细流转见 `userFlow.md`。

- 正常：完整走通 提问 → 过程流式展示 → 报告 路径
- 异常：SSE 断连自动重连，重连失败提示手动刷新且不丢已生成内容
- 边界：移动端浏览器可读（响应式），但不做移动端优化

#### FR-8 REST API（API First）

所有诊断能力通过 REST API 暴露，OpenAPI 3.0 文档随服务发布。Web UI 是 API 的第一个消费者，不是唯一消费者。v0.95 的 Webhook 端点契约在本版本一并定义（不实现）。

- 正常：仅通过 API（不开 UI）可完成完整诊断流程
- 异常：统一错误码规范（见 `apiDesign.md`）
- 边界：API 版本前缀 `/api/v1`，向前兼容策略在 `apiDesign.md` 定义

#### FR-9 Docker 一键部署

docker-compose 启动全部组件（Nginx + 应用 + PostgreSQL）。

- 正常：`docker compose up -d` 后 ≤10 分钟内可完成首次诊断（G3）
- 异常：端口冲突/环境变量缺失时启动日志给出明确指引
- 边界：支持 amd64 + arm64 镜像

#### FR-10 最小鉴权（安全需求，完整表述）

系统首次启动时生成单管理员账号的初始密码并强制修改；Web UI 使用会话登录；REST API 使用 Bearer Token（管理员可在设置页生成/吊销）。所有 LLM API Key 及 Prometheus/Elasticsearch 凭证在数据库中加密存储，任何 API 响应与日志输出中不得出现明文凭证。多用户与 RBAC 明确不在本版本范围（v2.0）。

- 正常：未认证请求一律 401；登录后所有功能可用
- 异常：连续登录失败触发限速；Token 吊销后立即失效
- 边界：初始密码仅在首次启动日志/首启页面出现一次

#### FR-11 SSE 流式输出

诊断是多步长过程（查询规划 → 取数 → 分析），过程状态与 LLM 输出通过 SSE 实时推送：用户能看到"正在查询 Prometheus…""发现异常区间…"等中间步骤。

- 正常：提问后 ≤5 秒出现首个过程事件
- 异常：流中断时已推送内容保留，可查询最终结果补齐
- 边界：单次诊断超时上限（草案 120 秒）后终止并输出已有结论

#### FR-12 LLM 多后端配置

设置页配置 LLM 提供商（OpenAI / Anthropic / DeepSeek / Ollama 等兼容后端）、模型名、API Key、Base URL。模型无关是架构约束（handoff 1.6）。

- 正常：切换后端后无需重启即生效；提供连通性测试按钮
- 异常：Key 无效/配额耗尽时错误信息可读，指向设置页
- 边界：本地 Ollama 无 Key 场景正常工作

## 10. 设计考量

- 交互范式：对话式（类 ChatGPT），过程透明是信任基础——中间步骤必须可见（FR-11），黑盒出结论不可接受
- 报告即产品：Markdown 报告是可分享物，是社区传播载体，视觉质量高优先级
- 页面清单与状态机 → `userFlow.md`
- 可访问性：键盘可完成核心流程、对比度达标；不做完整 WCAG 审计（B2D 工具，务实优先）
