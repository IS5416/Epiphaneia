# 测试阶段 (Testing)

## 目标

对开发完成的系统进行全面测试，确保功能正确、性能达标、用户体验符合预期，为上线提供质量信心。

## 输入

- `Development/` 完成的功能代码
- `ProductPlanning/` 的 PRD（验收标准）
- `ArchitectureDesign/` 的安全设计和部署架构

## 产出物

| 产出 | 文件名 | 说明 |
|------|--------|------|
| 测试计划 | `testPlan.md` | 测试范围、策略、环境、排期 |
| 集成测试报告 | `integrationTestReport.md` | 接口联调、数据一致性、第三方服务集成 |
| 性能测试报告 | `performanceTestReport.md` | QPS/TPS、响应时间、资源消耗、瓶颈分析 |
| 安全测试报告 | `securityTestReport.md` | 渗透测试、漏洞扫描、安全合规检查 |
| 验收测试报告 | `acceptanceTestReport.md` | 对照PRD逐项验收，通过/失败/备注 |
| 缺陷清单 | `bugList.md` | 所有发现的缺陷、严重等级、修复状态 |

## 行为约束

1. **独立环境**：测试在独立于开发环境的环境中进行，数据不与开发环境混合。
2. **性能基线**：接口 P95 响应时间 ≤ 500ms，P99 ≤ 1000ms（可根据业务调整）。
3. **安全必测**：必须覆盖 OWASP Top 10 相关漏洞测试。
4. **回归测试**：每次修复后必须运行完整回归测试套件。
5. **阻塞级缺陷**：P0 缺陷必须全部修复才能进入发布阶段；P1 缺陷允许有明确的修复计划。
6. **数据脱敏**：测试数据中不得包含真实用户信息。

## 退出条件

- 全部测试计划执行完毕
- P0 缺陷清零
- 性能指标达标
- 用户确认验收测试通过
