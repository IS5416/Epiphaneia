# 用户画像 — Epiphaneia

> 创建日期：2026-07-17
> 产品定位：面向后端开发者的 AI Agent 智能诊断工作台

---

## 三大核心用户画像

### 用户画像 A：后端开发者（主目标用户）

```
┌─────────────────────────────────────────────────┐
│                                                  │
│   👤 张浩 (Alex Zhang)                           │
│   28岁 | Java 后端开发 | 3-5年经验               │
│   某中型互联网公司（50-200人）                     │
│   技术栈：Spring Boot + MyBatis + K8s            │
│                                                  │
├─────────────────────────────────────────────────┤
│  每日工作                                         │
│  • 写 CRUD 和业务逻辑（主要时间）                   │
│  • 被拉去排查线上问题（最不想做的事）                │
│  • 看 Prometheus、翻 ELK 日志、追踪链路             │
│  • 参与 on-call 轮值，半夜被告警叫醒               │
│                                                  │
├─────────────────────────────────────────────────┤
│  核心痛点                                         │
│  😫 "半夜告警，看 Prometheus 只有'延迟高了'，       │
│      翻日志半小时才找到是某个慢查询"                 │
│  😫 "线上问题定位靠经验，新人不会排查"               │
│  😫 "事故报告要写半小时，每次格式还不一样"            │
│  😫 "同样的问题出现过 3 次，每次都重新排查"          │
│                                                  │
├─────────────────────────────────────────────────┤
│  使用场景                                         │
│  "为什么 user-service 的 p99 延迟从 200ms          │
│   涨到了 2s？" → Epiphaneia 自动查 Prometheus     │
│   指标 + ELK 日志 → 输出根因分析                   │
│                                                  │
├─────────────────────────────────────────────────┤
│  技术接触点                                       │
│  • 首选：Web UI（在公司电脑上用）                   │
│  • 其次：飞书/钉钉 Bot → 告警触发后直接推送诊断      │
│  • 不想装 CLI 工具                                 │
│                                                  │
├─────────────────────────────────────────────────┤
│  决策因素                                         │
│  • 免费开源是前提——小团队没有采购预算                │
│  • Docker 一键部署——不想折腾环境                    │
│  • 对现有监控工具零侵入——Prometheus/ELK 不能改      │
│                                                  │
└─────────────────────────────────────────────────┘
```

**关键洞察**：张浩代表的"后端开发者"群体是 Epiphaneia 的核心用户。他们的需求不是"缺少监控工具"（已经有很多），而是"监控数据太多、智能分析太少"。他们需要的不是另一个面板，是一个能替他们看数据、给结论的 AI 同事。

### 用户画像 B：DevOps/SRE 工程师

```
┌─────────────────────────────────────────────────┐
│                                                  │
│   👤 李思远 (Ryan Li)                             │
│   32岁 | SRE/DevOps | 6-8年经验                   │
│   某 FinTech 公司 | 管着 200+ 微服务               │
│   技术栈：K8s + Prometheus + ELK + Terraform      │
│                                                  │
├─────────────────────────────────────────────────┤
│  每日工作                                         │
│  • 盯着 Grafana 面板和告警                         │
│  • 处理基础设施问题（集群、网络、存储）               │
│  • 写 Terraform/Ansible 自动化脚本                 │
│  • 和开发"甩锅"——是代码问题还是基础设施问题？        │
│                                                  │
├─────────────────────────────────────────────────┤
│  核心痛点                                         │
│  😫 "告警太多了——90% 是噪音，但不敢关掉任何一个"     │
│  😫 "开发问我'为什么挂了'，我要查 3 个系统才能回答"   │
│  😫 "事故复盘报告最花时间，经常拖延"                  │
│  😫 "一个人管 200 个服务，根本看不过来"              │
│                                                  │
├─────────────────────────────────────────────────┤
│  使用场景                                         │
│  Prometheus 告警 → AlertManager webhook          │
│  → Epiphaneia 自动触发诊断 → 输出根因 + 建议      │
│  → 推送飞书群："user-service 延迟飙升，根因是      │
│     Redis 连接池耗尽，建议扩容从 10 → 30"          │
│                                                  │
├─────────────────────────────────────────────────┤
│  决策因素                                         │
│  • 需要 API First——方便集成到现有工具链             │
│  • 关注安全——不能自动执行危险操作                    │
│  • 诊断结果要可追溯——完整的时间线和证据链            │
│                                                  │
└─────────────────────────────────────────────────┘
```

**关键洞察**：Ryan 代表的是"运维专家"群体。他们是 Epiphaneia 的深度用户——提出的诊断问题更复杂（"为什么 kafka consumer lag 突然涨了 10x？"），但也是最有价值的社区贡献者（能够贡献诊断规则、Connector 插件）。

**关键洞察**：Maria 代表 Epiphaneia 的"扩展用户"——她的需求超出了 MVP 范围（团队视角、测试/审查/文档 Skill），但她是付费意愿最高的用户群体（企业版核心买家）。

## 用户画像总结

| 维度 | A: 后端开发者 | B: SRE/DevOps | C: Tech Lead |
|------|-------------|---------------|-------------|
| 核心需求 | 快速定位线上问题 | 自动化诊断 + 降噪 | 团队效率 + 质量治理 |
| 使用频率 | 有问题时才用（每周 2-3 次） | 每告警都触发（每日多次） | 周期性回顾（每周/月） |
| 技术深度 | 中等——看日志/指标够了 | 深——需要跨系统关联分析 | 中高——关注趋势和模式 |
| 付费意愿 | 低（个人） | 中（团队） | 高（公司预算） |
| MVP 覆盖 | ✅ 核心场景 | ✅ 核心场景 | 🟡 部分覆盖（单点诊断） |
| 扩散力 | ★★★★★ 社区传播主力 | ★★★★ 技术影响力 | ★★★ 管理决策者 |
