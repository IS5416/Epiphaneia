## 阻断级问题 (BLOCKER — 必须修复)

### B-1: SSE Publisher 线程不安全（变量在单例 Bean 上）
**来源**：设计一致性 + 架构安全审查
**文件**：`DiagnosisOrchestratorImpl.java:43,60-62`
**问题**：`ssePublisher` 是单例 `@Service` 的实例字段，通过 `setSsePublisher()` 设置。并发诊断会覆盖对方的发布器，导致跨会话 SSE 事件泄露。
**修复**：将 publisher 移入 DiagnosisContext 或作为 execute() 参数传入。

### B-2: COMPLETED_PARTIAL 降级路径完全缺失
**来源**：设计一致性审查
**文件**：`DiagnosisOrchestratorImpl.java:89-149`
**问题**：状态机允许 COMPLETED_PARTIAL，但编排器从未使用。数据源全部失败、无数据源时都直接进入 COMPLETED。
**修复**：添加 completePartial() 方法，queryingPhase 跟踪失败数，无成功查询时使用 COMPLETED_PARTIAL。

### B-3: QUERYING 阶段异常导致非法状态迁移 (→FAILED)
**来源**：设计一致性审查
**文件**：`DiagnosisOrchestratorImpl.java:99-104`
**问题**：通用 catch 调用 fail()，将状态设为 FAILED。但 FAILED 不能从 QUERYING 进入，只能从 PLANNING/ANALYZING。
**修复**：使异常处理感知状态——QUERYING 期异常 → COMPLETED_PARTIAL，PLANNING/ANALYZING 期异常 → FAILED。

### B-4: LlmClient 完全忽略 ModelRouter 和 LlmProvider
**来源**：设计一致性审查
**文件**：`LlmClient.java` + `DiagnosisOrchestratorImpl.java`
**问题**：LLM 调用未使用用户配置的 provider/model/apiKey——永远使用默认 ChatClient，FR-12 实际不可用。
**修复**：LlmClient 接受 LlmProvider，调用前用 ModelRouter 解密 API key 并配置 ChatClient。

### B-5: 原始异常消息泄露至 SSE 和数据库
**来源**：架构安全审查
**文件**：`DiagnosisOrchestratorImpl.java:103,230-239,147-155`
**问题**：`e.getMessage()` 被直接推送 SSE + 持久化 failureReason + 发送至 LLM（通过证据）。可能泄露 JDBC 连接串、主机端口、HTTP 响应体。
**修复**：SSE/持久化使用通用错误消息，详细信息限服务器日志；连接器异常不注入 LLM。

### B-6: ReportSynthesizerImpl 零测试
**来源**：测试质量审查
**文件**：`ReportSynthesizerImpl.java` (183 行，无测试)
**问题**：报告生成层零覆盖——LLM 路径、模板降级、空数据、null 字段，全部未验证。
**修复**：新增 `ReportSynthesizerImplTest`，至少覆盖 LLM 成功/失败降级/空消息/无 AGENT 消息。

### B-7: transition() 对 null diagnosisState 无防护
**来源**：设计一致性审查
**文件**：`DiagnosisOrchestratorImpl.java:245-246`
**问题**：`State.valueOf(null)` 抛出 NPE → 被通用 catch 捕获 → 设定为 FAILED，处理不当。
**修复**：将 null 诊断状态视为 CREATED。
