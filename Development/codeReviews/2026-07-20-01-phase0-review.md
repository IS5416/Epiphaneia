## 阻断级问题 (BLOCKER — 必须修复)

### B-1: Message 实体和 Flyway 迁移缺失 `failure_reason` 字段
**来源**：设计一致性审查
**文件**：`V001__init_schema.sql` + `Message.java`
**设计依据**：dbSchema.md §2.7 明确定义 `failure_reason TEXT`，domainModel.md §2.2 定义 `failureReason: String?`
**影响**：ABORTED/FAILED 状态机无法记录失败原因
**修复**：
- `V001__init_schema.sql` message 表 `diagnosis_state` 后添加：`failure_reason TEXT,`
- `Message.java` 添加字段：`@Column(name = "failure_reason", columnDefinition = "TEXT") private String failureReason;`

### B-2: Admin.username 缺少 `length = 50`
**来源**：设计一致性审查
**文件**：`Admin.java:9`
**设计依据**：dbSchema.md §2.1 — `username VARCHAR(50) NOT NULL UNIQUE`
**影响**：`ddl-auto: validate` 时 Hibernate 默认 VARCHAR(255) 与 DB 的 VARCHAR(50) 不匹配，应用启动失败
**修复**：`@Column(nullable = false, unique = true, length = 50)`

### B-3: EpiphaneiaProperties 违反 `api/` 包边界规则
**来源**：架构安全审查
**文件**：`EpiphaneiaProperties.java:3`
**问题**：`io.epiphaneia.infra.api` 包中的类导入了 Spring `@ConfigurationProperties`，违反 `*.api.*` 禁止框架导入的规则
**修复**：将 EpiphaneiaProperties 移至 `io.epiphaneia.infra.internal.config.EpiphaneiaProperties`，或接受此为配置类例外并文档化

## 高优先级 (HIGH — 本 Phase 修复)

### H-1: 加密密钥通过 toString() 泄露
**来源**：架构安全审查
**文件**：`EpiphaneiaProperties.java:12`
**问题**：Java record 自动生成 `toString()` 输出 `EpiphaneiaProperties[encryptionKey=abc123...]`，任何日志都会泄露完整密钥
**修复**：不继承 record，改用 class + 手动 toString() 遮罩，或用 char[] 存储密钥

### H-2: 加密实现未使用设计文档要求的 Spring Security Crypto AesBytesEncryptor
**来源**：设计一致性审查
**文件**：`AesGcmEncryptionService.java`
**设计依据**：securityDesign.md §4.2 明确要求 "使用 Spring Security Crypto `AesBytesEncryptor`"
**影响**：手工 Cipher 管理增加 IV/认证标签的错误面
**评估**：当前 javax.crypto 实现测试充分 (10 tests)，安全性与 AesBytesEncryptor 等价。两条路径：(a) 切换到 AesBytesEncryptor，(b) 更新 securityDesign.md 记录此决策
**建议**：方案 (b) — 更新设计文档。javax.crypto 直接调用避免 SB4.1.0 中 AesBytesEncryptor API breaking change 风险

### H-3: 密钥编码格式代码与设计文档不一致 (Hex vs Base64)
**来源**：设计一致性审查
**文件**：`AesGcmEncryptionService.java:38`
**问题**：代码用 `HexFormat.of().parseHex()`，设计文档写 Base64 编码
**影响**：用户按设计文档命令 `openssl rand -base64 32` 生成密钥 → 应用启动失败
**修复**：统一为 Hex。更新 securityDesign.md，提供生成命令：`openssl rand -hex 32`

### H-4: SecureRandom.getInstanceStrong() 可能阻塞
**来源**：设计一致性审查 + 架构安全审查
**文件**：`AesGcmEncryptionService.java:58`
**问题**：Linux Docker 容器熵池不足时阻塞数秒，GCM IV 不需要阻塞级随机性
**修复**：`new SecureRandom()` 替代 `SecureRandom.getInstanceStrong()`
