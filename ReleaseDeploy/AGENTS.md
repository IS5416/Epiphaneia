# 发布上线 (Release & Deploy)

## 目标

将测试通过的版本安全、平滑地部署到生产环境，并向用户交付可用产品。

## 输入

- `product/` — 测试通过的待发布版本（含 `docker-compose.yml`、Dockerfile、CI 配置）
- `Testing/` 的测试报告和缺陷清单
- `ArchitectureDesign/deploymentArchitecture.md` 的部署方案

## 产出物

| 产出 | 文件名 | 说明 |
|------|--------|------|
| 发布检查清单 | `releaseChecklist.md` | 上线前逐项确认（配置、证书、数据、回滚方案） |
| CI/CD 配置 | `cicd/` | 流水线配置文件（Jenkinsfile / GitHub Actions / GitLab CI） |
| 环境配置 | `envConfig/` | 生产环境变量清单（值脱敏，只保留key和说明） |
| 灰度发布策略 | `canaryStrategy.md` | 灰度比例、观察窗口、回滚触发条件 |
| 发布记录 | `releaseNotes.md` | 版本号、变更内容、已知问题、发布日期 |

## 行为约束

1. **回滚优先**：上线前必须有可验证的回滚方案，没有回滚方案禁止上线。
2. **灰度原则**：禁止一键全量，必须经过灰度发布逐步放量。
3. **数据备份**：上线前必须完成数据库完整备份并验证可恢复。
4. **变更窗口**：在低流量时段发布，提前通知相关方。
5. **可观测性**：确保监控、日志、告警在上线前已就绪。
6. **应急预案**：准备常见故障场景的处理脚本和联系人。
7. **审批门**：所有生产变更需用户最终确认，不可自动执行。

## 退出条件

- 发布检查清单全部通过
- 灰度各阶段观察窗口无异常
- 全量发布完成且监控指标正常
- 用户确认上线成功
